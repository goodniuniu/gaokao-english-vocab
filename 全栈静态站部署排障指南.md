# 全栈静态站部署排障指南

> **适用场景**：静态站（GitHub Pages）+ 自定义域名 + CDN 加速（EdgeOne）+ Serverless 后端（Cloudflare Worker）
> **来源**：高考英语单词练习项目的实际排障实践（2025-07）
> **状态**：知识库资源，持续更新

---

## 一、架构总览

```
用户浏览器
  ↓ HTTPS（TLS 握手对象 = CDN 边缘节点）
CDN 边缘节点（EdgeOne，需自己的边缘证书）
  ↓ HTTPS 回源（回源 Host 决定源站识别哪个站点）
静态站源站（GitHub Pages，自有证书）
  ↓ fetch 跨域请求（带 Origin 头）
Serverless 后端（Cloudflare Worker，检查 Origin 白名单）
  ↓
KV 存储（Cloudflare KV）
```

### 各组件职责

| 层 | 组件 | 职责 | 证书/安全 |
|---|---|---|---|
| DNS | 阿里云 | 域名解析 | — |
| CDN | EdgeOne | 国内加速 + HTTPS 终止 | **需自己的边缘证书** |
| 源站 | GitHub Pages | 托管静态资源 | GitHub 的 Let's Encrypt 证书 |
| 后端 | Cloudflare Worker | API + 数据 | Origin 白名单 + 限流 |
| 存储 | Cloudflare KV | 数据持久化 | 同步码即凭证 |

## 二、六个核心概念（每个都是踩过的坑）

### 概念 1：DNS 同一主机只能有一条 CNAME

**原理**：DNS 标准（RFC 1034）规定同一主机名不能有多条 CNAME 记录。DNS 服务器遇到多条时行为不确定（可能轮询、可能只返回一条）。

**坑**：同时配了 `www → GitHub Pages` 和 `www → EdgeOne CNAME`，以为两条都能用，实测 DNS 在两条之间轮询，导致用户每次访问解析到不同服务器，行为不稳定。

**正确做法**：同一主机只保留一条 CNAME。切换 CDN 时先确保新的通，再删旧的。

**验证命令**：
```bash
dig +short www.example.com CNAME    # 多查几次，确认只返回一条
```

---

### 概念 2：HTTPS 证书跟服务器走，不跟域名走

**原理**：SSL/TLS 证书安装在**服务器**上。同一个域名 `www.example.com`，两台不同的服务器各自需要装各自的证书。浏览器只跟它**直接握手的那台**服务器验证证书。

**坑**：以为 GitHub Pages 已经有 `www.example.com` 的证书，CDN 就不用再申请了。实际上 CDN 架构下浏览器跟 CDN 边缘节点握手，GitHub 的证书管不到这段。

**CDN 架构的两段 TLS**：
```
浏览器 ←TLS①→ CDN 边缘节点  ←TLS②→ 源站
         CDN需要证书          源站需要证书（各自的，独立的）
```

**正确做法**：CDN 必须有自己的边缘证书（免费 DV 即可）。不能白嫖源站的证书。

**验证证书归属**：
```bash
echo | openssl s_client -connect www.example.com:443 -servername www.example.com 2>/dev/null \
  | openssl x509 -noout -issuer -subject
```

---

### 概念 3：CORS 的 Origin = scheme + host + port

**原理**：浏览器跨域请求时自动发送 `Origin` 头，其值为 `scheme://host:port`，**不含路径**。后端做白名单时必须精确匹配。

**坑**：以下都被视为不同的 Origin，白名单必须分别添加：
- `https://example.com` ≠ `http://example.com`（协议不同）
- `https://example.com` ≠ `https://www.example.com`（www vs 裸域）
- `https://example.com` ≠ `https://example.com:8080`（端口不同）

**特殊值 `Origin: null`**：以下场景浏览器发送 `null`，会被白名单拦截：
- `file://` 协议打开（双击 HTML）
- iframe sandbox
- 浏览器严格隐私模式（Brave / Firefox Strict）
- data: URI

**正确做法**：白名单把所有可能的访问来源都列上（http/https × www/裸域）。如果需要支持 `file://`，设为 `*` 或单独处理 null。

**验证 Origin 是否放行**：
```bash
curl -sS -o /dev/null -w "HTTP %{http_code}\n" -X POST \
  -H "Origin: https://你的域名" \
  -H "Content-Type: application/json" \
  -d '{"name":"test"}' \
  https://你的API/api/endpoint
```

---

### 概念 4：CDN 回源 Host 的陷阱

**原理**：CDN 回源时发送的 `Host` 头决定了源站识别哪个站点。Host 配错会导致源站返回 404 或重定向循环。

**坑**：GitHub Pages 绑定了自定义域名 `www.example.com` 后：
- 回源 Host = `www.example.com` → ✅ GitHub 返回 200（正确识别站点）
- 回源 Host = `用户.github.io` → ❌ GitHub 返回 301（重定向到自定义域名 → 死循环）

**判断规则**：
- 源站**已绑定自定义域名** → 回源 Host 用**自定义域名**
- 源站**未绑定自定义域名** → 回源 Host 用**源站原始域名**

**验证回源**：
```bash
# 模拟回源 Host = 你的自定义域名
curl -sS -o /dev/null -w "HTTP %{http_code}\n" \
  -H "Host: www.example.com" \
  https://用户.github.io/项目路径/
# 200 = 正确，301 = 会循环，404 = 站点不存在
```

---

### 概念 5：浏览器自动升级 HTTPS

**原理**：现代浏览器（Chrome、Edge）默认开启 HTTPS-First，即使输入 `http://` 也会先尝试 `https://`。HSTS 缓存会强制 HTTPS。

**坑**：以为用户用 HTTP 访问就行，HTTPS 暂时坏了无所谓。实际上大量用户会被浏览器自动升级到 HTTPS，然后看到证书错误。

**正确做法**：HTTPS 必须保证可用，不能依赖"用户用 HTTP 就行"。

---

### 概念 6：CDN 强制 HTTPS 跳转 + 无证书 = 全挂

**原理**：CDN 开启"强制 HTTPS 跳转"后，HTTP 请求会被 301 重定向到 HTTPS。如果此时 HTTPS 证书还没就绪，用户走 HTTP → 301 → HTTPS → 证书错误 → 无法访问。

**坑**：证书签发中就开了强制跳转，导致全站不可访问。

**正确做法**：先确保 HTTPS 完全可用（curl 验证 200），再开强制跳转。

## 三、排障方法论：分层隔离法

遇到"访问异常"时，从下往上逐层排查：

```
第 1 层：DNS 解析对不对？
  ↓ dig 验证
第 2 层：TCP/TLS 连得上吗？证书对吗？
  ↓ curl / openssl 验证
第 3 层：HTTP 响应正常吗？
  ↓ curl -I 验证状态码
第 4 层：应用层逻辑对吗？（CORS / Origin / 认证）
  ↓ curl 带不同 Header 模拟
第 5 层：前端代码对吗？
  ↓ DevTools Network 面板看实际请求/响应
```

### 关键原则

1. **先确认服务端没问题，再查客户端**：用 curl 排除浏览器干扰
2. **逐个变量排查**：Origin、Host、协议、域名分别测试
3. **看实际请求**：DevTools Network 面板 > 猜测
4. **DNS 有传播延迟**：改了 DNS 后等 TTL 时间再验证（通常 600s）

## 四、常见问题速查表

### 问题 1：API 报 403

| 根因 | 排查 | 解决 |
|---|---|---|
| Origin 不在白名单 | DevTools 看 403 请求的 `Origin` 头 | 加入 Worker 白名单 |
| Origin: null（file:// / 隐私模式） | Origin 头值为 `null` | 白名单设为 `*` 或改用 https 访问 |
| 白名单改了没部署 | `wrangler deploy` | 重新部署 |

### 问题 2：HTTPS 时好时坏

| 根因 | 排查 | 解决 |
|---|---|---|
| DNS 多条 CNAME 轮询 | `dig` 多查几次，看是否返回不同结果 | 只保留一条 CNAME |
| CDN 证书未关联 | `openssl s_client` 看 subject | 关联 DV 证书到加速域名 |
| CDN 证书传播中 | 等 5~10 分钟 | — |

### 问题 3：CDN 回源 404

| 根因 | 排查 | 解决 |
|---|---|---|
| 回源 Host 配错 | `curl -H "Host: xxx"` 模拟测试 | 改成正确的回源 Host |

### 问题 4：证书域名不匹配

| 根因 | 排查 | 解决 |
|---|---|---|
| CDN 返回默认通配符证书 | `openssl s_client` 看 subject | 关联自己的 DV 证书 |

## 五、代码工程经验

### 5.1 错误提示的用户友好性

**问题**：后端返回英文错误（如 `Origin not allowed`），前端直接透传给用户，普通用户看不懂。

**解决方案**：前端增加错误映射层，按 HTTP 状态码 + 已知英文文案映射成中文：

```javascript
function friendlyError(status, data) {
  var raw = data.error || '';
  // 已知英文 → 中文
  var enToZh = { 'Origin not allowed': '访问被拒绝，请通过官方网站访问' };
  if (enToZh[raw]) return enToZh[raw];
  // 已是中文直接用
  if (raw && /[\u4e00-\u9fa5]/.test(raw)) return raw;
  // 按状态码给默认提示
  var statusMsg = {
    403: '访问被拒绝',
    404: '资源不存在',
    429: '请求过于频繁，请稍后重试',
    // ...
  };
  return statusMsg[status] || raw || ('请求失败 (HTTP ' + status + ')');
}
```

### 5.2 事件监听器生命周期管理

**问题**：`addEventListener` 用匿名函数，后续无法 `removeEventListener`。多次调用导致同一事件注册多个监听器。

**错误模式**：
```javascript
function start() {
  window.addEventListener('beforeunload', function() { ... }); // 匿名，无法移除
}
```

**正确模式**：
```javascript
var handler = null;
function start() {
  if (!handler) {
    handler = namedFunction;  // 命名引用
    window.addEventListener('beforeunload', handler);
  }
}
function stop() {
  if (handler) {
    window.removeEventListener('beforeunload', handler);
    handler = null;
  }
}
```

### 5.3 异步操作的容错

**问题**：`register` 成功后调 `uploadAll`，后者失败直接抛异常，用户以为注册失败。

**解决**：区分主操作和副作用的优先级，次要操作失败不阻塞主流程：

```javascript
try {
  await secondaryOperation();
} catch(e) {
  markForRetry();  // 标记待重试
  console.warn('次要操作失败，稍后重试:', e.message);
  // 不 throw，主操作不受影响
}
```

### 5.4 后端 JSON 解析保护

**问题**：`JSON.parse(text)` 遇到非法 JSON 抛异常，进 catch 后返回 500（服务器内部错误），但实际是客户端的错。

**解决**：先 try-catch 解析，失败返回 400：

```javascript
let body;
try {
  body = JSON.parse(text);
} catch(e) {
  return json({ ok: false, error: '请求体不是合法的 JSON' }, 400);
}
```

## 六、多平台配置速查

### 6.1 阿里云 DNS

| 操作 | 要点 |
|---|---|
| 主机记录 | 相对根域名，不写后缀（填 `www` 不是 `www.example.com.`） |
| CNAME 冲突 | 同一主机只能一条 CNAME，切换时先确保新的通再删旧的 |
| TXT 验证 | 主机记录同样不带域名后缀 |
| 暂停 vs 删除 | 暂停记录保留可恢复，更安全 |

### 6.2 EdgeOne CDN

| 配置项 | 要点 |
|---|---|
| 回源地址 | 源站域名（如 `用户.github.io`） |
| 回源 Host | 源站绑定的域名（自定义域名已绑就用自定义域名） |
| 回源协议 | HTTPS 或协议跟随 |
| 边缘证书 | 免费 DV，需**关联到加速域名**（不是"部署到云资源"） |
| 证书验证 | DNS 加 TXT 记录，等签发（通常 5 分钟~24 小时） |

### 6.3 Cloudflare Worker

| 配置项 | 要点 |
|---|---|
| ALLOWED_ORIGINS | 逗号分隔，精确匹配 scheme+host+port |
| 改配置后 | 必须 `wrangler deploy` |
| 限流 | KV 固定窗口计数，敏感接口加 |
| Origin 检查 | `if (origin && allowedOrigins !== '*')` —— null Origin 是 truthy 字符串 |

### 6.4 GitHub Pages

| 配置项 | 要点 |
|---|---|
| 自定义域名 | Settings → Pages → Custom domain |
| 域名验证 | 需加 TXT 记录验证裸域所有权 |
| HTTPS | Enforce HTTPS（GitHub 自动签 Let's Encrypt） |
| CNAME 文件 | 自动生成在仓库根目录 |

## 七、关键命令速查

### DNS 诊断
```bash
dig +short example.com CNAME          # 查 CNAME
dig +short example.com A              # 查 A 记录
dig TXT _dnsauth.example.com +short   # 查 TXT 验证记录
# 多次查询检测轮询
for i in 1 2 3 4 5; do dig +short www.example.com CNAME; done
```

### HTTPS/证书诊断
```bash
# 查证书签发方和主体
echo | openssl s_client -connect example.com:443 -servername example.com 2>/dev/null \
  | openssl x509 -noout -issuer -subject -dates

# 查证书覆盖的域名（SAN）
echo | openssl s_client -connect example.com:443 -servername example.com 2>/dev/null \
  | openssl x509 -noout -text | grep -A1 "Subject Alternative Name"

# curl 测试 HTTPS
curl -sS -o /dev/null -w "HTTP %{http_code}  SSL:%{ssl_verify_result}  %{time_total}s\n" \
  https://example.com/
```

### API 诊断
```bash
# 带不同 Origin 测试 CORS
curl -sS -o /dev/null -w "HTTP %{http_code}\n" -X POST \
  -H "Origin: https://example.com" \
  -H "Content-Type: application/json" \
  -d '{"name":"test"}' \
  https://api.example.com/endpoint

# 模拟不同回源 Host
curl -sS -o /dev/null -w "HTTP %{http_code}\n" \
  -H "Host: www.example.com" \
  https://源站域名/
```

## 八、架构决策参考

### 何时用 CDN（EdgeOne/Cloudflare），何时直连源站

| 场景 | 推荐 | 理由 |
|---|---|---|
| 面向国内用户，大流量静态资源 | CDN | 国内节点加速 |
| 自己用 / 小范围用 | 直连 | 省去 CDN 配置复杂度 |
| 后端 API（小数据量） | 直连 Worker | API 延迟可接受，省 CDN 配置 |

### 何时用全 Cloudflare，何时混合多平台

| 方案 | 优点 | 缺点 |
|---|---|---|
| 全 Cloudflare（DNS+Pages+Worker） | 最简单，一平台搞定 | 国内访问质量一般 |
| GitHub Pages + EdgeOne + Worker | 国内访问快 | 配置复杂，多平台 |
| GitHub Pages 直连 + Worker | 配置最简 | 国内慢（~1s） |

**选择原则**：
- 优先简单（直连 / 全 Cloudflare）
- 国内速度有要求才加 CDN
- 后端 API 不需要 CDN（数据量小，延迟可接受）
